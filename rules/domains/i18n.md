## Internationalization Domain Rules

- [WARN] EXTERNALIZE STRINGS: All user-facing strings must be extracted to resource/translation files. No hardcoded English strings in components or templates.
- [WARN] UTF-8 ENCODING: All source files, resource bundles, and data interchange must use UTF-8 encoding. Declare encoding explicitly in HTML meta tags, HTTP headers, and database connections. Never assume ASCII.
- [WARN] INTL APIS: Use `Intl.DateTimeFormat`, `Intl.NumberFormat`, and `Intl.Collator` (or equivalent) for locale-aware formatting. Never manually format dates, numbers, or currency.
- [STRICT] RTL SUPPORT: Use logical CSS properties (`margin-inline-start`, `padding-inline-end`) instead of physical (`margin-left`). Test layouts in RTL languages (Arabic, Hebrew).
- [STRICT] ICU FORMAT: Use ICU MessageFormat for all translatable strings that involve plurals, gender, or select expressions. ICU handles complex grammar rules across languages that simple string interpolation cannot.
- [STRICT] NO STRING CONCATENATION: Never build translated strings by concatenating fragments. Use parameterized messages with named placeholders. Word order varies between languages.
- [STRICT] LOCALE FALLBACK: Define a fallback chain for missing translations (e.g., `fr-CA` -> `fr` -> `en`). Never show translation keys to users. Alert on missing translations.
- [STRICT] DATE AND TIME ZONES: Store dates in UTC. Convert to user's timezone for display. Always show timezone indicator for ambiguous times. Use IANA timezone identifiers.
- [CRITICAL] TRANSLATION COMPLETENESS: Block release if translation coverage falls below threshold (95%+). CI must validate all locale files parse correctly and have no missing keys.
- [CRITICAL] TEXT EXPANSION: Design UI to accommodate text expansion (German ~30% longer than English). Never truncate translated text. Test with pseudo-localization to catch layout issues early.
